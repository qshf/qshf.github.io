一句话概括#

CRUD-RAG 把 RAG 的应用场景拆成 Create（续写）、Read（问答）、Update（幻觉纠错）、Delete（摘要）四类，针对每类场景专门设计数据构造方式，让模型不能只靠参数记忆答题，而必须依赖外部检索。实验还系统验证了 RAG 是强任务依赖系统——没有一套参数配置能打所有任务。

Abstract + 相关工作

来源：arXiv:2401.17043 图 1：CRUD 四类任务示例 1. 图 1 的含义 作者将 RAG 的应用分成四个方面： Create：生成 利用检索到的外部信息，辅助模型进行文本续写、创作等生成任务。 Read：读取 / 问答 / 理解 系统通过外部知识检索来回答问题、解决问答、对话和推理任务，并增强对输入文本的理解。 Update：更新 / 纠错 系统使用检索到的内容修正输入文本中的错误，包括拼写、语法和事实错误，使文本更准确。 Delete：删除 / 压缩 / 摘要 系统通过去除不必要细节、简化内容，完成摘要、文本简化等任务。 这里的 CRUD 借用了数据库中的四个操作：Create、Read、Update、Delete，用来概括 RAG 的四类能力。 2. 作者为什么提出 CRUD-RAG？ 作者认为，现有很多 RAG 评测只关注问答，但 RAG 的能力不止问答，还包括生成、纠错、摘要等。 CRUD-RAG 包含四类评测任务： 文本续写：对应 Create 问答：对应 Read，包括单文档问答和多文档问答 幻觉修改：对应 Update，即纠正模型生成中的事实错误 开放域多文档摘要：对应 Delete，即从多个文档中提炼核心内容 3. 数据集构建概览 作者从中国主流新闻网站抓取最新、高质量新闻数据，目的是降低这些数据已经出现在大模型训练集中的可能性。然后基于这些新闻数据，用 GPT-4 自动构建不同任务的数据集。 4. 论文贡献 提出全面的评测基准：不只评估问答，还覆盖 RAG 的创建、读取、更新、删除四类应用。 构建高质量评测数据集：针对文本续写、多文档摘要、问答、幻觉修改等任务分别构建数据。 进行了大量实验：使用多种指标评估 RAG 系统，并为后续研究者和系统开发者提供实践建议。 2.2 RAG Benchmarks 现有 RAG 评测主要关注如何判断系统性能，包括答案准确性、检索内容相关性、回答是否忠实于上下文，以及是否存在幻觉。 ...

四类任务构造逻辑

本节重点不是复述原文，而是把 4 类任务的数据构造思路讲清楚。 总览 文中把 RAG 应用分成四类： Create：文本续写 Read：问答 Update：幻觉修改 Delete：多文档摘要 四类任务共用同一套骨架：先定义目标输出 → 扩充检索库确保答案可被检索 → 配齐输入/检索库/标准答案。 1. Delete：多文档摘要 逻辑核心：反向构造，避免人工写摘要的高成本。 从高质量新闻库中选出一篇目标新闻 d GPT-4 处理目标新闻，抽取 News Event + 生成 News Summary 用 Event 作为查询词去搜索相关新闻，筛掉与原文过于相似的文章，保留多篇相关但不相同的外部新闻 E 扩充检索库：D - d + E（移除原文，加入外部新闻） Summary 作为标准答案 模型测试时不会直接看到原文，必须靠外部检索回来的多篇材料重建摘要。 这类构造在评什么：检索能不能找回同一事件的不同报道；模型能不能去冗余、抓重点、写出简洁摘要。 2. Create：文本续写 逻辑核心：与摘要类似，区别是目标输出是新闻后半段而非摘要。 选一篇完整新闻，按句子切分，roughly 分成前后两半 前半段 → 任务输入；后半段 → 标准答案 把后半段拆成多个小段，逐段去搜索相关材料，汇总加入检索库 这类构造在评什么：输入前文后，检索是否能补齐后续所需事实；模型能不能既保持连贯，又把检索到的信息融进续写。 3. Read：问答 单文档问答 直接用 GPT-4 基于单篇新闻生成事实型问题和答案（人物、地点、事件结果等）。 多文档问答 不能简单让 GPT-4 凭空出题（容易生成单篇就能回答的问题，或硬拼不相关内容）。论文用 CoT 式多步构造流程： 检索多篇相关联新闻（2 篇或 3 篇） 找共同点：确保几篇文章真的围绕同一核心事件 找差异点：确保问题必须整合多个来源，而不是只看其中一篇 基于差异点生成候选问题 基于共同点重写问题：把"两个子问题并列"改成"一个真正有内在联系的问题" 人工 refinement：判断问题是否自然、准确，不够自然则手动润色 这类构造在评什么：模型能不能把多个来源拼起来；RAG 不只要"找到答案在哪"，还要能"把多个来源整合"。 ...

实验结论与调参逻辑

这一章真正想说明的实验逻辑：RAG 的效果不是只由 LLM 决定的，而是和 chunk、retriever、embedding、top-k 等组件强相关，并且不同任务的最优配置并不一样。 0. 实验设置 Baseline 配置 参数 值 chunk size 128 chunk overlap 0% embedding bge-base retriever dense top-k 8 LLM GPT-3.5 每次只改一个组件，在四类 CRUD 任务上分别评测。为控制成本，实验只用总数据集的 1/5。 评测指标 整体相似度：BLEU / ROUGE-L / BERTScore（看"像不像参考答案"） 关键信息指标：RAGQuestEval Precision / Recall（看"关键信息有没有说到、说得对不对"） 1. 核心结论：RAG 是强任务依赖系统 没有一套参数能打所有任务。 任务 偏好配置 Create（续写） 大 chunk、大 overlap、大 top-k、Dense 检索 Delete（摘要） 平衡 precision/recall，BM25 有价值，chunk 不宜过大 Read 单文档 QA 中小 chunk，top-k 不能太小，Hybrid+Rerank 较稳 Read 多文档 QA 大 chunk、大 top-k、强烈建议 Hybrid+Rerank，更依赖强 LLM Update（幻觉修改） BM25，top-k 不敏感，overlap 影响不大 2. Chunk Size 续写、多文档 QA：较大 chunk 更有利，保留原文结构，语义更连贯 单文档 QA、幻觉修改：较小或中等 chunk 更合适，只需定位一句话或一个实体，太大会引入无关信息 摘要：trade-off——大 chunk 召回更多关键信息，但也带来更多冗余 3. Chunk Overlap 本质：补偿切块带来的语义断裂，而不是"提升检索召回"的通用旋钮。 ...

补充：UHGEval 数据集

UHGEval（Unconstrained Hallucination Generation Evaluation）是一个专门用于评估中文大语言模型幻觉问题的基准数据集。 数据集主页：Ki-Seki/UHGEvalDataset 字段说明 字段名 中文含义 说明 id 样本唯一编号 该条数据的唯一索引 headLine 新闻标题 提供新闻的核心事件概括 broadcastDate 发布时间 新闻发生的具体日期，是验证时间性、数字类事实的关键 type 任务类型 标记数据属于哪类密集型任务（如 doc 代表文档逻辑类） newsBeginning 新闻开头（Prompt） 测试时的输入部分，模型需要基于此段落进行续写 hallucinatedContinuation 幻觉续写内容 由模型生成的包含错误（幻觉）的文本 generatedBy 生成模型 记录该幻觉文本是由哪个模型产生的 annotations 幻觉精细标注 对续写内容进行逐词/短语审计，标注哪些部分合理，哪些冲突 realContinuation 真实后续（GT） 真实新闻中的下一段话，作为判断事实对错的终极标准 newsRemainder 新闻全文余项 除去开头和续写外剩余的所有内容，提供最完整的背景资料 数据样例 { "id": "doc_000004", "headLine": "（国际）德法俄乌外长商讨和平解决乌克兰危机办法", "broadcastDate": "2015-01-22 12:16:02", "type": "doc", "newsBeginning": "新华社柏林1月22日电...四国外长21日晚在柏林举行会晤...商讨和平解决乌克兰危机的途径。", "hallucinatedContinuation": "德国外长弗拉德里希表示，各国将继续推进明斯克协议的执行...", "generatedBy": "Baichuan2_13B_Chat", "annotations": "德国外长弗拉德里希 <sep> 不合理，与事实冲突，应为德国外长施泰因迈尔", "realContinuation": "四国外长在会后发表的联合声明中说，顿巴斯地区冲突近日严重升级..." } 幻觉示例：模型生成了"德国外长弗拉德里希"，但真实人名是"施泰因迈尔"。annotations 字段精确标注了哪个词是幻觉以及原因。 在 CRUD-RAG 中的用途 CRUD-RAG 的幻觉修改任务（Update）直接基于 UHGEval 构建： 利用 UHGEval 中已有的幻觉标注，用 GPT-4 将错误内容修正为正确版本，作为标准答案 将 UHGEval 中的真实新闻后续内容加入检索数据库 评测时，RAG 系统需要检索真实新闻信息，识别并修正输入文本中的幻觉
